# Plat — notes for Claude Code

食べて美味しかった料理を「写真＋メモ」だけで手早く記録する個人用PWA。姉妹アプリ[[Sake（表示名Rove）]]をベースに作った（`../Sake/CLAUDE.md`参照）が、ジャンル・評価・価格などの項目は持たず、写真とリッチメモの2つだけに絞っている。

## これはWebアプリ（PWA）。ネイティブ機能は使っていない

`index.html`だけで完結する純粋なブラウザアプリ。`file://`で直接開いても動くが、Service Workerの登録とPWAインストールはhttps/localhostのようなsecure contextでないと機能しない。

## secure context依存API（2026-07-20、実機LANテストで踏んだ実バグ）

`crypto.randomUUID()`は**secure context（https、または`localhost`）でのみ存在する**。開発中に同一Wi-Fi経由でスマホ実機から`http://<LAN-IP>:port/`（例: `http://192.168.1.4:8766/`）にアクセスして確認したところ、`crypto.randomUUID is not a function`のエラーが発生し、その1行の失敗が原因で**初期化処理全体が途中で止まり**「メモ欄に文字入力できない（`<input>`自体が生成されない）」「写真クロップのドラッグが効かない（`pointerdown`リスナー登録処理まで到達しない）」という一見無関係な2つの不具合が同時に起きた。原因はすべて`window.addEventListener('DOMContentLoaded', async () => {...})`内の`currentBlocks = [newBlock()]`が同期的にthrowし、それ以降の`renderEditor()`・クロップキャンバスへの`addEventListener`・`switchTab('input')`・SW登録が軒並み実行されなかったこと。

**重要な教訸（今後のテスト方針に直結）**：`localhost`は`http://`でもブラウザの仕様上secure context扱いになる特例があるため、`http://localhost:8766/`でのCDP検証ではこの種のバグを**原理的に再現できない**。実機を同一Wi-Fi上のLAN IPで検証して初めて発覚した。今後このアプリ（および他のPWA）で「secure context依存のAPIが動くか」を確認したい場合は、`localhost`ではなく実際のLAN IPやデプロイ先URLでテストする必要がある。

対策として`genId()`ヘルパー（`crypto.randomUUID`があれば使用、なければ`Math.random()`ベースのUUID風文字列にフォールバック）を追加し、`newBlockId()`と`doAutosaveNow()`内の直接呼び出しを置き換えた。本番のGitHub Pagesデプロイ後（https）は`crypto.randomUUID`が使われ続けるので実質的な変更はなく、あくまでLAN実機テストや将来の非https環境での耐障害性を上げるための修正。

他に`navigator.share()`もsecure context依存だが、こちらは元から`if (!navigator.share)`のガードがあり、この場合はエラーにならず「共有機能に非対応」のトーストが出るだけなので対応不要（今回のLAN実機テストで正常にフォールバックすることも確認済み）。

## エラートーストのフィルタ（2026-07-20、実機テストで発覚）

`window.addEventListener('error', ...)`はページ上のあらゆるスクリプトエラーを拾うため、**自アプリのコードとは無関係な要因（ブラウザ拡張機能の注入スクリプト、広告/トラッカーブロッカーなど）のエラーまで拾ってしまう**。実機テストで原因不明の「エラー: Script error.」というトーストが出た（`message`にファイル名・行番号を伴わない空疎な文字列 = ブラウザが「別オリジンのスクリプトなので詳細を開示しない」ときの標準の代替文字列）。自アプリのインラインscript内の本物のエラーは常に具体的なメッセージを持つため、`e.message === 'Script error.'`の場合だけトーストを出さずに握りつぶすようにした。これによって本物のバグを見逃すことはない。

## メモ入力中のキーボード対策（2026-07-20、一度実装してユーザー判断で撤回した経緯あり）

姉妹アプリ（Sake/Rove）から移植した「フォーカス中は`body`の高さを完全凍結する」方式だけでは、Platの伸び続けるブロック型メモには不十分ではないか、と考えて一時`freezeMainHeightForKeyboard()`（`<main>`自体をキーボード表示後の実際の可視高さまで縮めて自動スクロールする仕組み）を追加した。CDPでの検証は通ったが、実機で確認したユーザーから「Roveと同様、背景は一切動かさずキーボードがそのまま上に乗るようにしてほしい」と明確な差し戻し指示があり、**その仕組みは全面的に撤回した**。

現在の実装は姉妹アプリと完全に同じ、「フォーカス中は`body`の高さを凍結するだけで、それ以外は一切何もしない」方式（`syncViewportHeight()`／`fieldIsFocused`のみ）。Platのメモは行が増えるとキーボードの裏に隠れて見えなくなる場面が起こり得るが、これは把握した上でのユーザーの意図的な選択（「動かないこと」を「常に全行が見えること」より優先）。**今後同じ「入力中に文字が見えない」系の要望が来ても、`<main>`の動的リサイズやスクロール追従を安易に再実装しないこと** — 一度実装して明確に却下された選択肢である。

## エラーフィルタの強化（2026-07-20、2回目の実機テストで発覚）

上記の「Script error.」フィルタだけでは不十分だった。実機で「エラー: ReferenceError: Can't find variable: __firefox__」というトーストが出た——`__firefox__`はこのアプリのコードのどこにも存在しない識別子で、ブラウザ（またはブラウザ拡張機能）がページのコンテキストに注入するブリッジ的なグローバル変数と見られる。この種のエラーは`message`が空疎な"Script error."ではなく**具体的な文言を持つ**ため、既存のフィルタをすり抜けていた。

このアプリは単一のインラインscriptしか持たないため、自アプリの本物のエラーは常に`e.filename === location.href`になるはず、という原理に基づき、`e.filename`が存在してかつ自ページのURLと一致しない場合もフィルタするよう強化した（`e.filename`が空の場合はフィルタしない＝fail open。情報不足で本物のバグを握りつぶすよりは、多少ノイズが漏れる方を許容する設計）。CDPで5項目検証（外部由来のエラーは`filename`があっても弾かれる／`filename`なしのエラーは今まで通り表示される／自ページ発の本物のエラーは表示される／既存の"Script error."フィルタも回帰していない）、全てOK。

## データの持ち方

- IndexedDB（DB名 `platlog`、ストアは`items`のみ）。**DB名・`localStorage`キーは姉妹アプリ（`sakelog`）と意図的に別名にしてある** — IndexedDBはパス単位ではなくオリジン単位で共有されるため、同じGitHub Pagesオリジン配下で名前が被ると相手の実データを読み書きしてしまう。今後ストア/キーを増やす際もこの原則を忘れないこと。
- 写真は**トリミング後の正方形（800×800 JPEG q0.85）と、トリミング前の元画像（長辺1600px・JPEG q0.9にリサイズ済み）の両方**をBlobで保持し、`crop`にズーム/パン値も保存する。姉妹アプリは元画像を保持せず再トリムのたびに劣化する設計だったが、Platでは「トリミング位置をあとで変更できる」という要件を優先し、元画像から都度フル画質で再トリムできるようにした（ストレージ増分は個人用途なら許容範囲と判断）。
- メモは、**自前のブロック型データ構造**（`{id, type, text, checked}`の配列）として保存する。`type`は`paragraph`/`heading`/`bullet`/`checklist`の排他選択。各ブロックは独立した`<input>`で、Enter/Backspaceによる分割・結合も自前実装。**contenteditable/execCommandは意図的に不使用** — iOS Safariのcontenteditableは実機でのカーソル位置バグが多いことが姉妹アプリ開発の経験則としてあり、この方式を選んだ（ユーザーとの事前合意事項）。
- **太字/下線/取り消し線（文字単位の装飾）は2026-07-20のFB1で全廃止した。** 当初はブロック単位の真偽値フラグ（`bold`/`underline`/`strike`）として実装していたが、ユーザーから「選択した文字だけに適用したい」という要望が出た際、`<input>`ベースのままでは実現できず（`<input>`は一部だけ太字にするような混在スタイルを表示できない）、選択範囲対応にはcontenteditable化が必要になるという設計上のトレードオフを提示した。ユーザーの回答は「文字装飾は全廃止！カーソルバグがでるのが一番嫌なので」— contenteditable回避を最優先する判断のもと、部分装飾機能ごと削除した。**今後また"部分装飾がほしい"という要望が出ても、contenteditable化を安易に提案しないこと** — 一度明確に却下された選択肢である。
- 検索用に、装飾情報を除いたプレーンテキスト（`memoText` = 各ブロックの`text`を`\n`で連結したもの）を保存時にキャッシュしている。全文検索はこのキャッシュに対する`Array.filter`＋`includes`のインメモリ線形探索（IndexedDBのFTSは使っていない・個人用途の件数なら十分）。
- バックアップはJSON書き出し/読み込み。**`{app:"plat", schemaVersion:1, ...}`を必ず含む** — 読み込み時に`app==='plat'`でない（姉妹アプリのバックアップ等）を弾く。写真・元画像ともBase64化してJSON内に埋め込む。読み込みは項目ごとの`id`をそのまま`put()`でupsertするため、同じファイルを複数回読み込んでも重複しない（追加/置換どちらのモードでも）。

## 自動保存の設計（他アプリと異なる大きな特徴）

- 入力タブは「新規下書き」と「既存アイテムの編集」を同じ状態変数（`currentItemId`/`currentBlocks`/`currentPhotoBlob`等）で扱う。**内容が1文字でもあれば`currentItemId`が発行され、以後は600msデバウンスで`doAutosaveNow()`が直接IndexedDBに書き込む** — 明示的な「保存」操作なしに永続化される。
- 入力タブを離れる瞬間（`switchTab`／`visibilitychange`）に、内容が空なら（新規下書きだった場合のみ）DBから該当レコードを削除して幽霊アイテムを残さない。既存アイテム編集中に空にした場合も同様に削除される。
- 編集開始時（`startEdit`）に**編集前スナップショット**（`editSnapshot`）と**戻り先のbrowseStack**（`editingReturnStack`）を保存する。「リセット」は下書き新規作成時は単純クリアだが、既存アイテム編集中は**スナップショットをDBに書き戻して本当に元に戻す**（自動保存が既にDBへ反映済みのため、フォームをクリアするだけでは元に戻らない）。
- **重要な既知の罠（一度実際に踏んで修正済み）：** 編集の完了（保存/リセットどちら経由でも）では、`currentItemId`だけでなく`currentBlocks`/`currentPhotoBlob`等の入力タブの生存状態も必ず`resetInputForNewEntry()`でクリアすること。`currentItemId`だけクリアして生存状態を残すと、次に入力タブを何もせず再訪しただけで「新規下書きに中身がある」と誤認され、直前に保存/破棄したはずの内容が**別IDの幽霊アイテムとして複製保存される**（CDPテストで実際に再現・修正した実バグ）。

## Service Workerのキャッシュ名とクリーンアップ（モノレポ特有の注意）

キャッシュ名を`plat-`プレフィックス付き（`plat-v1`）にし、`activate`時の削除処理も`plat-`で始まるキーだけを対象にしている。**姉妹アプリ（Sake/Rove）のService Workerは全キャッシュキーのうち自分のCACHE_NAME以外を無条件削除するロジックのため**、同一オリジン上でプレフィックスを付けずに運用すると、どちらかのSWが更新されるたびに互いのオフラインキャッシュを消し合う（IndexedDBの実データには影響しないが、オフライン起動が壊れる）。今後このリポジトリに新しいPWAを追加する際は、この`plat-`のようなプレフィックス方式を踏襲すること。

## GitHub Pagesでのデプロイ（サブパス配信の注意）

姉妹アプリと同じ理由で、`manifest.json`・`service-worker.js`・アイコンの参照はすべて相対パス。デプロイ先は姉妹アプリとは別サブパス（例: `https://yasuharashigeru.github.io/AI-work/Plat/`）を想定。

## 検証方法

- [[feedback_browser_app_cdp_testing]]の手法（`python -m http.server`＋ヘッドレスChrome＋生CDP、Node組み込みの`WebSocket`グローバルでドライバスクリプトを書く）で、クロップ・ブロックエディタ・自動保存・検索/ハイライト・編集/リセット・削除+アンドゥ・バックアップ書き出し/読み込み（`DOM.setFileInputFiles`で実ファイルを`#importFileInput`に注入し、本物の`change`イベント経由で`onImportFileChosen`を通す）を自動検証済み（計35チェック、すべてOK）。
- バックアップの書き出し検証は、ヘッドレスChromeの`<a download>`クリックによる実ファイルダウンロードが`Browser.setDownloadBehavior`経由でも`state:"canceled"`になり不安定だったため、`exportBackup()`から`buildBackupPayload()`を分離し、ダウンロードUIを介さずペイロードの中身を直接検証する方式に切り替えた。実機でのダウンロード動作そのもの（ファイルが実際に保存されるか）は未確認 — Chromeのbase実装なので基本的に動くはずだが、姉妹アプリ同様、iOS Safari実機での最終確認は別途必要。
- 未確認（実機Safari固有の挙動、ヘッドレスChromeでは代表できない）：カメラ/フォトライブラリからの写真選択、クロップの実際の指の動き、`navigator.share()`の実際の共有シート表示、ホーム画面追加後のアイコン表示。

## 未実装（v1時点、意図的なスコープ外）

- 文字単位（太字/下線/取り消し線）の装飾 — 2026-07-20のFB1でユーザーの明示判断により機能ごと廃止（上記参照）。見出し/箇条書き/チェックリストという「行の種類」の書式のみ提供する。
- 「最近削除した項目」（保持期間つきゴミ箱） — 確認ダイアログ＋直後5秒アンドゥのみ。トースト表示中にページを閉じると復元不可（姉妹アプリと同じ制約）。
- 1件共有はプレースホルダーAI画像や合成カード画像ではなく、`navigator.share({files:[写真], text:プレーンテキスト化したメモ})`のみ。非対応端末では「共有機能に対応していません」のトーストを出すだけ。

## 見出し/箇条書き/チェックの操作（2026-07-20 FB3で長押しポップアップ化）

FB2で常時表示のツールバーに統一した直後、「ボタンの存在感が違和感ある」というフィードバックがあり、`.memo-toolbar`（常時表示の3ボタン行）自体を廃止した。代わりに、各ブロックの`<input>`を550ms長押しする（`onBlockPointerDown`/`onBlockPointerMove`/`onBlockPointerUp`、移動許容10px）と、押した位置の近くにトースト同様の濃色バブル（`#formatPopup`）が浮かび上がり、見出し/箇条書き/チェックを選べる。選択すると即座にポップアップは閉じ、それ以外の場所をタップ（`#formatPopupOverlay`）しても閉じる。普段は完全に非表示（DOM上に存在するがdisplay:none）。

**実機未検証の既知リスク**：iOS Safariの`<input>`は長押しで**ネイティブのテキスト選択UI（拡大鏡・選択ハンドル）も同時に出ようとする**可能性があり、これは`user-select`やCSSでは確実に抑制できないWebKitの既知の制約。今回はこの自前ポップアップとネイティブUIが競合しないかを実機で確認できていない（ヘッドレスChromeでは代表できない領域）。競合が実際に問題になった場合の代替案：長押し対象を`<input>`本体ではなく行のラップ`<div class="memo-block">`側に付け替える（テキスト選択自体が起きない領域で長押しを検出する）。

「ボタン類が角丸の四角でごちゃごちゃ見える」「保存のオレンジ色が暑苦しい」というフィードバックを受け、複数ボタンが並ぶ箇所を軒並み「個別の角丸ボックス＋色塗り」から「区切り線だけで分けたフラットな行」に変更した：`.memo-toolbar`（見出し/箇条書き/チェック）、`.form-actions`（リセット/保存 — 保存も塗りを廃止しリセットと同じ見た目、太字のみで区別）、`.detail-actions`（編集/共有/削除 — 削除は赤いテキスト色のみで区別、背景色は付けない）、`.item-action-btn`（確認モーダル・読み込みモード選択モーダル — カード下端まで角丸を保ったまま`overflow:hidden`でボタン行を角丸に沿わせている）。設定シートの「バックアップを書き出す/読み込む」も専用の`.settings-btn`（塗りつぶし角丸）を廃止し、既存の`.sel-row`（罫線区切りのフラットな行、写真アクションシート等で既に使用）に統合した。**今後このアプリに新しいボタン群を追加する際は、個別に角丸＋背景色のボックスにするのではなく、この「線区切りのフラット行」パターンを踏襲すること。**

## 入力欄のインデント（2026-07-20 FB1で修正）

見出し/段落は接頭辞スペースを持たず行の左端にフラッシュで表示、箇条書き/チェックリストのみマーカー分の余白（`.memo-block-prefix`、幅20px+gap6px）を持つ。以前は全ブロックタイプに空の`<span class="memo-block-prefix"></span>`を挿入していたため、見出し/段落も無意味にインデントされてしまっていた（`blockRowHTML()`参照）。閲覧側の詳細画面（`detailBlockHTML()`／`.detail-memo-block`系CSS）は元から見出し/段落フラッシュ・箇条書き/チェックリストのみインデントの設計になっており、今回のバグは入力エディタ側だけの問題だった。
